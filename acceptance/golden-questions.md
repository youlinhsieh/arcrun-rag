# Golden Questions — pilot 驗收模板（rag-wave1 T7）

> 對應 design.md §7、tasks.md T7、requirements.md US1（"✅=pilot 客戶 20 題 golden questions 命中率達標，門檻與客戶共定"）。
> 用法：pilot 導入時，總管/業務與客戶**共定 20 題**（真實會問的問題，答案要能在客戶知識庫裡找到），填進下表 → 安裝完成後逐題查詢 → 人工判命中 → 算命中率 → 出報告。
> 第一波**人工跑**（人讀查詢結果、人判對錯），自動化（腳本批次跑+比對）留待第二波題量變大或多客戶重複跑時再做。

## 使用時機

1. **選題**（pilot 前，與客戶開會現場或訪談）：挑客戶內部真的會查的問題，涵蓋不同主題/深淺，優先挑「有明確答案文件」的（方便判對錯），也留幾題「答案分散在多文件」的（測 RAG 而非純關鍵字搜尋的價值）。
2. **首灌後、pilot 開跑前**：知識庫已完成 T3+T4+T8（收集端＋轉檔＋首灌）跑分，確認庫已就緒再跑 golden questions（跑太早只會測出「還沒灌資料」，不是查詢品質問題）。
3. **每次重大變更後回歸**（provider 換源、ingest 邏輯改動）：重跑同一份題目，比對命中率有無退步。

## 三模式（US1）

Portal 查詢面提供**關鍵字／語意／圖**三種模式（design.md §2 Portal 拆分後的搜尋介面）。跑分時每題記錄「哪個模式命中」——這本身是產品洞察：若某類問題只有語意模式命中、關鍵字模式落空，代表客戶知識庫用詞與提問用詞不一致，是精耕/同義詞層要處理的信號，不是查詢引擎的鍋。

## 題目表（範本，pilot 時填實際 20 題）

| # | 問題 | 預期答案摘要 | 答案所在文件（來源溯源用） | 提問類型 | 查詢模式 | 命中？(✅/❌) | 備註 |
|---|------|------------|------------------------|---------|---------|--------------|------|
| 1 | （範例）公司請假規定是什麼？ | （範例）填答案摘要 | （範例）HR手冊.md | 事實查找 | 關鍵字 | | |
| 2 | | | | | | | |
| 3 | | | | | | | |
| ... | | | | | | | |
| 20 | | | | | | | |

**欄位說明**：
- **提問類型**：事實查找／流程步驟／跨文件綜合／模糊語意（分類供事後分析「哪類問題弱」）。
- **查詢模式**：實跑時試哪個模式先命中就記哪個；若三模式都試過才命中，記全部試過的模式＋最終命中的那個。
- **命中？**：人工判斷——答案摘要與查詢結果是否一致（允許用詞不同，語意對即算命中）；來源溯源是否正確指向客戶真實文件也算判準之一（溯源錯＝即使答案對也算部分失敗，備註寫清楚）。

## 跑分與報告

```
命中率 = 命中題數 / 20
```

- 報告至少含：命中率、命中率門檻（與客戶共定，US1 驗收條件）、未命中題目的失敗原因分類（庫沒收錄該文件／收錄了但精耕沒萃取到關鍵資訊／查詢詞與文件用詞不匹配／三模式都試過仍落空）。
- 門檻未達標：對照失敗原因分類決定下一步（補收文件 vs 調精耕 prompt vs 調查詢邏輯），不是無差別重跑。

## 自動化（第二波，非本輪）

第一波故意不自動化：20 題量小，人工跑比寫自動化腳本快，且「人工判對錯」本身能發現自動化字串比對會漏掉的語意正確但措辭不同的情況。待有多個客戶重複跑同款流程、題量規模上升，再考慮：批次查詢腳本（打三模式端點）＋LLM-as-judge 初篩（人工複核，不全自動信任）。
